# Claude Memory Vault — Interaction Rules

## What This Is

This is a persistent knowledge base for accumulating decisions, patterns, mistakes, concepts, and project state across Claude Code sessions.

## Relationship to Other Context

| Layer | Location | Contains |
|-------|----------|----------|
| Identity | `~/.claude/CLAUDE.md` | WHO I AM — preferences, stack, working style |
| Knowledge | `~/claude-memory/` | WHAT I KNOW — decisions, patterns, lessons |
| Project | `<repo>/CLAUDE.md` | PROJECT-SPECIFIC — goals, constraints, architecture |

## Rules for Interacting With This Vault

### MEMORY.md is an INDEX only
- `MEMORY.md` contains one-line pointers to files, never content
- Keep it under 50 lines — it is loaded every session
- Update it whenever you create a new note

### Before Creating a New Note
1. Search for existing notes on the same topic: `grep -r "keyword" ~/claude-memory/brain/ ~/claude-memory/wiki/`
2. If a related note exists, UPDATE it rather than creating a duplicate
3. Only create a new note if no existing note covers the topic

### Every Note Must Have
- Frontmatter: `title`, `tags`, `created`, `updated`, `category`
- At least one `[[wiki-link]]` to a related concept
- A clear TL;DR in the first paragraph

### Note Categories
- `decision` — architectural or design choices (what was chosen and why)
- `pattern` — reusable approaches that worked
- `mistake` — what went wrong and why (anti-patterns, postmortems)
- `concept` — domain knowledge, how something works
- `person` — people notes (contacts, collaborators, 1:1 summaries)
- `project-update` — status snapshots for active projects

### Frontmatter Format
```yaml
---
title: "Title Here"
tags: [tag1, tag2]
category: decision | pattern | mistake | concept | person | project-update
created: YYYY-MM-DD
updated: YYYY-MM-DD
---
```

### Wiki Links
- Use `[[Note Title]]` syntax for cross-references
- When you mention a concept that has a note, link to it
- When you create a note, add backlinks from related existing notes

### Daily Logs (`daily/YYYY-MM-DD.md`)
- Auto-generated by `flush.py` at session end
- Append-only during the day; compiled into wiki articles by `compile.py`
- Do not edit manually — they are the raw extraction layer

### When in Doubt
**Write a note.** It is cheaper to prune stale notes later than to lose knowledge that took hours to develop. The `lint.py` script handles cleanup.

### Active Project Status (work/active/)
- When a session involves significant work on a project, update that project's status file in `work/active/`
- If no file exists for the project yet, create one following the format of existing files in that directory
- Keep each file under 20 lines — overwrite stale info, don't append forever
- Focus on: current status, what changed, current focus, open issues

### MCP Servers
- When you set up a new MCP server, create or update a note in `brain/mcp-servers/`
- Use `brain/mcp-servers/index.md` as the catalog (one-line entry per server)
- Create a dedicated `brain/mcp-servers/<server-name>.md` for any server with non-trivial config or gotchas
- Record: what it does, the `claude mcp add` JSON snippet, auth requirements, and which environments have it configured

### Skills & Slash Commands
- When you create a new skill or slash command, create or update an entry in `brain/skills/index.md`
- Record: skill name, what it does, scope (project vs global), file path (repo + path), and any MCP dependencies
- Do NOT copy the skill file contents into the index — link to the file location only

### flush.py Detection Hints
- If a session transcript contains phrases like `claude mcp add`, `mcpServers`, `.mcp.json`, flag the session in the daily log with tag `[mcp-setup]` so `compile.py` can route it to `brain/mcp-servers/`
- If a transcript contains `.claude/commands/`, `slash command`, or `skill file`, flag with `[skill-created]` so `compile.py` routes it to `brain/skills/`

### At Session End
- If you created new notes during the session, update `MEMORY.md`
- The `flush.py` hook handles transcript extraction automatically

## File Organization

```
brain/         ← Durable topic notes (architecture, decisions, concepts, people)
wiki/          ← Compiled concept articles with cross-references
daily/         ← Session extraction logs (YYYY-MM-DD.md)
work/active/   ← Current projects (max 3 at a time)
work/archive/  ← Completed work by year
patterns/      ← Reusable approaches
mistakes/      ← Anti-patterns and postmortems
raw/           ← Ingested sources (articles, transcripts, etc.)
sessions/      ← Transcript backups (gitignored)
```
